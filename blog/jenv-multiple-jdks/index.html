jEnv - Managing Multiple Jdks

Jul 26 2020

Introduction

Before Java 8 there was no need to have multiple JDKs on your machine. Java development process was slow enough that even big and rusty corporations could catch up. But since JDK 9 the development process is much faster and new Java versions are released more often.

Why even bother?

Some projects may run on an older JDK due to various reasons:

  • the tooling does not yet support new JDKs,
  • some libraries do not work on new JDKs - this can be seen when migrating from pre-JDK 9 to JDK 9 or onward,
  • sometimes it is better to wait than to be an early adopter for core production services,
  • and many, many more.

On the other hand being a good Java developer requires one to be up to date with updates introduced by new Java versions.

Where is the problem?

Every Java developer knows that java is only one of programs shipped with JDK. There are other important tools like javac, keytool, jps and many others. They can be often picked up by build tools like Gradle or Maven. Even if you set JAVA_HOME environment variable for target JDK, a tool might pick up another one and the build fails.

What is the solution?

Ideally we should be able to use the right JDK for the right project. Changing JAVA_HOME or PATH environment variables is tedious but fortunately there is a tool for that - jEnv

Usage

Just follow instructions on the web page. This tool is very simply to use yet very powerful. One thing that is not mentioned on the web page is that you can put .java_version file in the root directory of your project. This file should contain the JDK that you want to use for this project. The JDK version must match one of the values printed out by jevn versions command.

Moreover, you want to make sure that jenv doctor passes all checks.

Conclusion

Good luck and have fun with playing with various JDK whether it is for work or just for the pleasure of discovering new features in new JDKs.
